## Policy Summary

WE OUT HERE uses a hybrid workflow:

- development is primarily remote
- prep is mixed remote and in-person
- production is on-the-ground with remote support where useful
- post is coordinated remotely with scheduled live reviews

Remote work is not informal work. The same standards for clarity, respect, confidentiality, and accountability apply.

## Core Rules

- use approved project channels for project work
- keep important decisions in searchable written form
- do not rely on memory, private side chats, or disappearing messages for critical approvals
- respect time zones and work windows
- protect scripts, cuts, and personal information

## Communication Expectations

### Async first for non-urgent work

Use written updates for:

- status checks
- routine notes
- file handoffs
- simple approval requests

Use live meetings for:

- creative review that needs discussion
- schedule or budget decisions with multiple stakeholders
- conflict resolution
- sensitive conversations involving people issues

### Response-time norms

- urgent production or delivery issue: as quickly as possible through the designated urgent channel
- normal business-day question: same day or next business day
- non-urgent notes and review comments: within the agreed review window

## Working Hours And Time Zones

- each team member should share their primary working time zone and normal working hours
- do not schedule recurring meetings outside shared overlap unless necessary
- if a team spans more than 3 time zones, the meeting burden should rotate where practical
- silence outside working hours should not be treated as poor commitment

## File And Version Rules

- store project files only in approved shared locations
- use consistent file naming and version labels
- avoid duplicate working documents unless there is a clear reason
- final approvals should be reflected in file names, trackers, or written signoff notes
- never send sensitive final materials through unsecured personal channels if an approved alternative exists

## Review And Approval Rules

- every review request should state what is being reviewed, by when, and who decides
- if feedback is advisory rather than approval-required, label it clearly
- avoid scattered notes across multiple channels when one consolidated note pass will do
- unresolved conflicting notes should route to the actual decision-maker, not stay trapped in chat

## Meeting Norms

- default meeting length is 25 or 50 minutes
- share an agenda for meetings expected to last more than 20 minutes
- end with owners and next steps
- cancel meetings that no longer have a purpose

## Security And Confidentiality

- use least-access-needed permissions
- do not forward scripts, cuts, budgets, or contact details outside approved groups
- lock or remove access when someone wraps or leaves the project
- report suspected leaks, account compromise, or accidental over-sharing immediately

## Remote Conduct

The code of conduct applies fully in remote settings.

That includes:

- chat channels
- video calls
- review notes
- email
- text messages used for project work

Snide side-channel behavior, pile-ons, or humiliating note styles are not acceptable just because they happen online.

## Hybrid Production Support

During production, remote contributors should know:

- who the on-set decision-maker is for their issue
- when real-time responses are needed
- where final documents are posted
- how emergency changes will be communicated

Remote support should simplify production, not create parallel chains of confusion.

## Failure Modes To Avoid

- approvals living only in DMs
- same file being revised in multiple places with no owner
- late-night note dumps treated as normal workflow
- remote staff being excluded from context they need to succeed
- on-set teams waiting on remote decisions that were not properly scheduled

## Escalation

- workflow confusion -> direct manager or coordinator
- file access or tool issue -> ops support or assigned admin owner
- repeated after-hours pressure or disrespect -> manager, HR contact, or policy reporting path
- security issue -> ops or security owner immediately
