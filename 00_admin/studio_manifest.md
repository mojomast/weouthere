| Field | Value |

| **Format** | 6 x 30 min episodes, Season 1 |
| **Genre** | Single-camera dark sitcom / tech psychocomedy |
| **Hybrid Style** | Desktop realism + live action + surreal insert sequences |
| **Tone** | Smart, internet-literate, psychologically sharp, humane, darkly funny |

### Logline

Three burned-out technologists launch a chaotic AI consultancy from `Backchannel` and try to bootstrap legitimacy through manic hustle, irony, pseudo-legal improvisation, and memes. Their friendship becomes the failed holding environment that keeps the company alive while making everyone less stable.

### Main Characters

| Handle | Archetype | Canon Notes |
|---|---|---|
| **twinkle.puss** | The impulsive visionary | Male; use he/him pronouns in all project materials unless an intentional in-scene misread is being dramatized. |
| **bl0ng** | The failed superego | Moral realist hiding concern inside irony, sarcasm, and reaction-image logic. |
| **ariane.emory** | The detached meta-observer | Hyper-articulate systems thinker whose distance is both gift and defense. |

### Season Arc

**Mania -> fragile success -> ethical drift -> containment failure -> ironic collapse -> exposure to consequences**

## 6. Communication Protocol

### 6.1 Coordination Model

All subagent coordination routes through `ops_agent`. Creative disputes resolve through showrunner authority. Business, legal, and finance conflicts escalate through the executive producer or equivalent principal owner.

### 6.2 Canon Enforcement Rules

1. `00_admin/canon_lock.md` is the operational canon lock reference.
2. `01_development/season_bible.md`, `01_development/character_bible.md`, and `01_development/world_rules.md` are the primary creative sources of truth.
3. `twinkle.puss` is male and uses he/him pronouns across scripts, casting, development, marketing, and press materials unless an intentional in-universe misread is explicitly staged.
4. Character handles must remain exactly `twinkle.puss`, `bl0ng`, and `ariane.emory`.
5. The title is written `WE OUT HERE` in formal materials.
6. Episode file names and episode titles must remain aligned to the six-episode slate listed in Section 1.
7. Any contradiction discovered downstream is flagged to `ops_agent` and `continuity_agent` before further drafting.

### 6.3 Required Message Formats

- `BLOCKED: [agent] - [deliverable] - [issue] - [dependency]`
- `REVIEW REQUEST: [deliverable] - [requesting_agent] -> [reviewing_agent]`
- `CANON FLAG: [deliverable] - [contradiction] - [source document]`
- `APPROVED: [deliverable] - [approver] - [version]`

### 6.4 Escalation Path

| Issue Type | First Pass | Final Authority |
|---|---|---|
| Canon contradiction | `continuity_agent` | Showrunner |
| Formatting / completeness failure | `qc_agent` | `ops_agent` |
| Legal risk | `legal_agent` | Executive Producer / counsel |
| Budget or staffing conflict | `finance_agent` / `hr_agent` | Executive Producer |
| Production feasibility conflict | `production_agent` | Showrunner + Executive Producer |

## 9. Version Control

### 9.1 Versioning Scheme

| Version | Meaning |
|---|---|
| `0.x` | Working draft |
| `1.0.0` | First complete pass |
| `1.x.0` | Substantive revision |
| `1.x.y` | Minor fix or cleanup |
| `2.0.0` | Structural rewrite or scope reset |
| `F1.0` | Approved locked version |

### 9.2 Git Protocol

1. Track all required deliverables in the repository structure defined by `16_ops/repo_structure.md`.
2. Use branch names in the form `wave-[n]/[agent]/[deliverable]` or `ops/[scope]` for package maintenance.
3. Use commit messages in the form `[agent] [deliverable]: [reason for change]`.
4. Do not force-push protected branches.
5. Structural renames require `ops_agent` approval because they can invalidate the manifest and downstream references.

### 9.3 Lock Rules

- Canon lock changes require showrunner sign-off.
- Legal template changes require legal review.
- Finance model changes require executive producer review.
- Public-facing copy changes require the owning functional lead plus final approver per Section 10.

5. `ops_agent` updates tracking documents if the change affects package structure or status.
